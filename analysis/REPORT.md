# 医疗预问诊大模型 · SFT→DPO→GRPO 全链路实验报告

> 从零构建一个中文医疗预问诊助手:数据工程 → 监督微调(SFT) → 直接偏好优化(DPO) → 可验证奖励强化学习(GRPO/RLVR)。
> 基座 Qwen3-8B。硬件:早期 H20(公司共享,免费,做超参扫描)+ 后期 4×RTX 5090(租赁,做全量与 RLHF)。
> 本报告 + 全部图表/脚本/日志存档于 `F:\rlhf_lab_cloud_kit\`。配套详细时间线见 `docs/CLOUD_RUN_LOG.md` 与 `docs/EXPERIMENT_LOG.md`。

---

## 摘要

本项目端到端实践了大模型对齐的完整 RLHF 链路。核心成果与发现:

1. **数据工程**:从 217 万条医疗语料的备选池,经"统一schema→清洗→task_type分类→跨源去重→配比采样"管线,构建出 **10 类任务均衡、安全感知**的 SFT 数据集 v1.1(154,476 条),业务种子数据未被开源数据淹没。
2. **SFT 的关键发现(基座与模板匹配)**:用 **Qwen3-8B-Base(裸基座)** 训对话,即使充分训练(loss 1.055)仍**无法学会停止(倒豆子:一次生成多轮+乱码)**;换 **Qwen3-8B-Instruct + qwen3_nothink 模板**后**彻底解决**。证明问题在基座对齐与模板,不在数据。
3. **DPO**:用 SFT 模型自采样 + 双 API(MiniMax/DeepSeek)裁判构造 1,434 条偏好对(含长度偏置防护),beta 扫参选 0.3,将心理危机应对从"冷漠问诊"改善为"共情+引导专业帮助"。
4. **GRPO(RLVR)**:以 CMExam 选择题"答对=1"为规则奖励,四卡 400 步,**CMExam 独立评测集 62.6% → 65.5%(+2.9pts)**,完成长度稳定于 2–5 token,**无 reward hacking**,且不损害对话行为。
5. **方法论**:小验证再放量、多维度评测(交互测试不可替代)、reward hacking 监控、基座-模板匹配——全程实践并有数据支撑。

## 1. 数据工程

**目标**:面向医疗 Agent(导诊/预问诊/问答/报告解读/健教/RAG/结构化摘要),构建**受控、任务均衡、安全感知**的 SFT 数据集,而非盲目堆量。

**管线**(脚本见 `scripts_cloud/sft_data/`):
`01_convert`(6源→统一messages schema)→ `02_clean`(广告SEO/无警示开药/无检查下定论/PII/超短长/危险建议 规则过滤)→ `03_label`(规则分类 10类 task_type + risk/red_flags 标注)→ `04_dedup`(跨源精确+MinHash近重去重)→ `05_sample`(以业务种子 15,463 条为锚点、×0.5154 缩放 300k 模板做配比采样)。

**评测保留区红线**:CMB / CMExam / MLEC / CBLUE **禁入训练集**(否则基线作废);CMExam 另按 seed=42 切"RL训练 5,590 / 评测 1,000",永不重叠。

**最终 v1.1 = 154,476 条**(含后期补充的 323 条心理危机样本 ×2 过采样)。分布见 **图5**:任务上症状咨询/健康百科/分诊为主,来源上真实问诊(med_zh)+ 华佗百科 + 业务飞轮种子 + teacher生成(MiniMax-M3)多源混合。

![图5 数据分布](figures/fig5_data_distribution.png)

**核心理念**:不最大化数据量,而最大化**任务覆盖、安全行为、业务相关性**——让模型成为更好的医疗 Agent,而非更大的医疗 QA 记忆库。

## 2. 监督微调(SFT)

### 2.1 超参扫描(R0–R3,H20 免费卡)
40k 子集 × 1 epoch,单变量对照(rank 8/16/32/64 → lr 5e-5~4e-4 → 挂载层 → seed 方差)。**结论**:
- rank 越大越好(64 未饱和);lr 甜点 = 2e-4(U型曲线);all-linear 优于仅注意力层(16σ);
- **seed 噪声底 σ≈0.0007**——由此确立"loss 差 >0.003、CMB 差 >2.5pp 才算真效应"的判读标准;
- **重要副发现**:lr 过大(4e-4)时 dev loss 仅 +0.014,但 **CMB 从 73.8% 暴跌到 68.3%(灾难性遗忘)**——选 lr 必须同时看域内 loss + 域外知识保持。

### 2.2 数据配比对照(R4,四卡并行)
四臂 A纯开源/B纯种子/C混合/D种子2x,同超参、唯数据不同(**图1**)。CMB 知识保持:B纯种子 73.8%(=base 最佳)/ C混合 73.3% / A纯开源 72.5% / D 71.7%(**图6a**)。洞察:纯种子知识零损失但缺安全/RAG(那些是生成数据),**混合最均衡**。

![图1 R4四臂loss](figures/fig1_r4_arms_loss.png)

### 2.3 基座路线的关键转折(本项目最深的教训)
R5 用 **Base 基座**全量 154k×2ep,最终 loss 降到 1.055(充分收敛),但交互测试暴露**严重倒豆子**:一次生成多轮问答+结论、`�`乱码、复读。逐一排除假说:欠训练(loss已低,证伪)、停止符配置(证伪)、`<think>`数据污染(全量0%,证伪)。**根因锁定:Base 基座无对话对齐,学不会 `<|im_end|>` 停止;qwen3 模板注入 `<think>` 致乱码。**

**解法**:换 **Qwen3-8B-Instruct(已具对话/停止机制)+ `qwen3_nothink` 模板**。40k 小验证即**倒豆子 4/4 单轮停住、0 乱码**。全量正式版 loss 更低(图2),交互流畅。

![图2 Base vs Instruct](figures/fig2_sft_base_vs_instruct.png)

> **原理**:预训练基座(Base)只学"续写",未建立 `<|im_end|>` 的"对话轮结束"语义;SFT 要从零教这个特殊 token 的用法,数据量再大也难稳。Instruct 版经官方对齐已内建此机制,SFT 只需注入领域行为。**当初选 Base 是为 CPT,但 CPT 实测对已强 base 无提升而放弃 → SFT 就应该用 Instruct。**

## 3. 直接偏好优化(DPO)

**造数据**:用 SFT 模型对业务 prompt 每题采样 K=4 候选,**MiniMax-M3 + DeepSeek 双 API 裁判**排序取最好/最差组 chosen/rejected,**内置长度偏置防护**(chosen 比 rejected 长 >2.5x 则丢弃,避免"越长越好"污染)。得 **1,434 条偏好对**。

**beta 扫参**(0.1/0.3/0.5,图3):beta 越小越放开优化、越大越贴近参考。三版差异小(DPO 稳定)。人工对比各版实际输出后**选 beta=0.3**(问诊追问最自然、心理共情、其它均衡)。

![图3 DPO beta扫参](figures/fig3_dpo_beta_loss.png)

**效果**:心理危机场景从 SFT 的"平静问持续多久"改善为 DPO 后的"听到你这么说我很担心你…我在这里陪着你"(共情+引导热线)。

> **原理**:DPO 用闭式偏好损失直接优化"chosen 概率 > rejected 概率",beta 控制与参考模型的 KL 约束。相比 PPO 无需训奖励模型、无需在线采样,稳定且省显存。

## 4. 可验证奖励强化学习(GRPO / RLVR)

**为何 GRPO 不选 PPO**:单卡/共享卡放不下 PPO 的 4 个模型;GRPO 用**组内相对优势**替代 critic,省一个模型。

**奖励设计(RLVR)**:CMExam 医学选择题,模型生成答案→抽取字母→与标准答案比对,**答对=1 答错=0**(规则可验证,无需奖励模型)。起点 = DPO(0.3) 合并后的模型。四卡 400 步。

**信号(图4)**:reward 平滑线 0.60→0.68 上升;**完成长度稳定 2–5 token**(模型只答字母,未靠灌水骗奖励);KL≈0.0003 受控;熵下降(更自信)。**全程无 reward hacking**——这是 RL 最大风险,通过监控 reward+长度确认规避。

![图4 GRPO reward](figures/fig4_grpo_reward.png)

**效果(图6b)**:CMExam 1000 题独立评测集 **DPO 62.6% → GRPO 65.5%(+2.9pts)**。且五场景复测确认**对话行为无损**(RL 强化选择题未致"顾此失彼")。

> **原理**:GRPO 对同一 prompt 采样一组 completion,以组内奖励均值为基线算相对优势,做策略梯度 + KL 惩罚。RLVR 用规则奖励(答案可判)替代人类偏好/奖励模型,是数学/代码/选择题等"可验证任务"的高效对齐方式。工程正解应用 **vLLM 做 rollout**(推训分离),本项目 GRPO 用 transformers 生成(短答案够快),vLLM/veRL 路线为后续工业级进阶。

## 5. 方法论总结(可迁移的经验)

1. **小验证再放量**:超参/方向用小数据(40k/1ep)便宜地定,再上全量。避免了多次 8 小时全量白跑(如 base-vs-instruct 用 40k 验证即定论)。
2. **多维度评测,交互测试不可替代**:脚本五场景全绿,但用户两句真实对话("你好"/"肚子疼")即暴露幻觉+跳步;规则评分器一度把复读/乱码判满分。**loss/选择题测不出生成退化与对话鲁棒性**。
3. **reward hacking 监控**:GRPO 全程盯 reward + 完成长度,长度不暴涨才敢采用。
4. **基座-模板匹配**:本项目最深教训——chat SFT 用 Instruct 基座 + 匹配模板,不用裸 Base。
5. **统计纪律**:先测 seed 噪声底,再判断效应是否真实(否则 0.3% 的"提升"可能是噪声)。
6. **可复现规范**:seed/optimizer/模板/停止符全字段显式记录;评测保留区与训练集严格隔离。

## 6. 已知局限与后续方向

**当前模型的能力边界(用户亲测发现)**:
- 分诊科室不够准(腹泻→误推外科),根在种子数据科室标注质量;
- 跑题不纠偏、首轮闲聊会幻觉症状——缺"越界拒绝/开场引导/纠偏澄清"样本;
- 多角色混训导致边界模糊。

**架构演进方向(与工业界主流一致)**:从"单一全能 SFT"进化为 **Router + 共享基座多 LoRA 专才 + 对话状态机**:
- 前端/router 按角色(预问诊/健康百科/报告解读/健康闲聊)分入口;
- 同一 Instruct 基座热切换各角色 LoRA(部署省显存,单角色独立迭代);
- 每角色独立的越界拒绝护栏("宫保鸡丁"/"维生素B"在预问诊里拒答并导流);
- 预问诊挂**槽位填充状态机**(harness 层管"当前收集主诉/既往史/过敏史",LLM 负责填空),解决跑题问题。
- v1.1 数据已按 task_type 分类,天然可拆成各角色训练集,不需重标。

**工程进阶**:veRL / vLLM 推训分离重跑 GRPO(路线B,独立环境;5090 sm_120 对 vLLM 版本要求严格,需 torch 2.8/cu128 匹配,记录见 `docs/VERL_SETUP_LOG.md`)。
